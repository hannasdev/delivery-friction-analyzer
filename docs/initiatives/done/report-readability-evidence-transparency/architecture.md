# Report Readability And Evidence Transparency Architecture Notes

## Context

The analyzer now has a working live data path:

1. collect GitHub source data;
2. normalize it through repository profile rules;
3. compute `friction-metrics.v1`;
4. render `friction-report.v1` JSON and Markdown;
5. attach collection coverage to the live report artifacts.

The next initiative changes the report and artifact presentation layer. It should improve readability and transparency without moving data collection, normalization, metric calculation, or recommendation logic into the CLI.

## Current State

- `src/cli/analyze-github.js` orchestrates collection, normalization, metrics, report generation, Markdown rendering, artifact writes, and stdout summary.
- `src/report/friction-report.js` generates `friction-report.v1` JSON and Markdown from `friction-metrics.v1`.
- `src/report/generate-report.js` generates JSON and Markdown from an existing metrics summary for fixture and advanced workflows.
- Existing full analysis artifacts are:
  - `source-bundle.json`
  - `normalized.json`
  - `metrics-summary.json`
  - `friction-report.json`
  - `friction-report.md`
- Live analysis currently appends collection coverage Markdown from the CLI layer after the report renderer returns Markdown.
- CLI stdout currently prints a long JSON completion receipt by default.

## Target Shape

The report layer should own human-facing report structure and evidence presentation. The CLI should own orchestration, option parsing, progress output, artifact transaction safety, and live collection coverage attachment.

The full analysis artifact set should expand to include:

- `methodology.md`
- `pr-metrics.csv`
- `bottleneck-examples.csv`
- `comment-sources.csv`
- `collection-coverage.csv`

CSV artifacts should be generated by default for full runs and suppressed by `--no-csv`. The long completion JSON should be printed only with `--json`.

The Markdown report should remain the primary human-facing artifact. CSVs and JSON are supporting evidence trails.

## Decisions

| Decision | Rationale | Alternatives Considered |
| --- | --- | --- |
| Generate CSV exports by default with `--no-csv` opt-out. | Spreadsheet-friendly evidence should be available without a second command, while sensitive users can suppress it. | Opt-in `--csv`, or only JSON artifacts. |
| Keep raw JSON artifacts. | JSON remains the deterministic contract for tests and future UI work. | Replace JSON with CSV or Markdown-only output. |
| Add detailed `methodology.md` plus a concise methodology summary in the report. | Readers need quick context in the report and deeper explanation when reviewing trust boundaries. | Inline full methodology in every report, or only maintain a static docs page. |
| Print concise CLI output by default and move full receipt to `--json`. | Maintainers need the report path more than a large receipt. Machine consumers still have an explicit mode. | Keep current JSON stdout, or write no structured receipt at all. |
| Treat sensitivity analysis as explanatory context, not filtered ranking. | Outliers are real data, but readers need to know when a conclusion is fragile. | Remove outliers automatically, or leave only the existing dominance note. |
| Keep CSV exports curated. | Transparency should not mean exporting raw comments or secret-bearing data in a spreadsheet-friendly form. | Dump normalized data to CSV wholesale. |
| Generate `methodology.md` from a hybrid renderer. | Most methodology text should be stable, but run-specific facts make the artifact trustworthy for a particular analysis. | Static repository doc only, or fully dynamic prose generated from report data. |
| Deduplicate sensitivity by dominant PR. | Several bottlenecks can share the same outlier, and repeating the same exclusion story would add noise. | Only analyze the top bottleneck, or compute separate duplicate summaries for every bottleneck. |

## Contracts And Boundaries

- Metrics contract:
  - `friction-metrics.v1` should remain the source for scores, counts, and rankings.
  - This initiative should not change formulas unless a bug is found.
- Report JSON contract:
  - `friction-report.v1` should continue separating observed data, inferred diagnosis, suggested action, dominance, coverage, guardrails, and follow-up.
  - Any new sensitivity summary should be represented explicitly if it is included in JSON.
  - Backwards-compatible optional fields can remain under `friction-report.v1`; removing or renaming existing fields should require a report-version bump.
- Markdown contract:
  - Markdown should present report data for maintainers, not introduce untraceable claims.
  - The renderer should keep observed facts and inferred text visibly distinct.
- CSV contract:
  - CSVs are curated evidence exports derived from existing report, metrics, and coverage data.
  - CSVs must not include raw comment bodies, token values, environment details, or individual ranking outputs.
  - CSVs should be deterministic and stable enough for tests.
  - Minimum column groups:
    - `pr-metrics.csv`: PR identity, title, URL, diff counts, review counts, validation counts, iteration counts, and ranking scores.
    - `bottleneck-examples.csv`: bottleneck identity, recommendation category, PR identity, score/value, validation evidence, review evidence, comment-source counts, dominance, and source labels.
    - `comment-sources.csv`: source name, total comments, classification flags, and share of all comments.
    - `collection-coverage.csv`: API family, status, attempts, source label, diagnostics, and downstream impact.
  - Empty CSV cells mean unavailable or not applicable; numeric zero should be used only for observed or computed zero counts.
- CLI boundary:
  - CLI should orchestrate artifact generation and print progress/completion status.
  - CLI should not compute metric formulas or embed report-specific interpretation prose.
  - In `--json` mode, stdout should contain only parseable JSON. Progress output and caveats should move to stderr or be suppressed.
- Sensitivity boundary:
  - Sensitivity analysis should call existing metric computation over a filtered in-memory sample or equivalent report-layer helper.
  - Sensitivity analysis should not alter baseline metrics, source bundles, or normalized artifacts.
  - The baseline report remains authoritative; sensitivity summaries explain robustness and outlier dependence.
- Artifact transaction boundary:
  - New artifacts must participate in the existing safe-write behavior so failed runs do not leave a mix of old and new outputs that appear complete.

## Migration / Compatibility

Existing users will still get the core artifacts. New default CSV and methodology artifacts add files but do not remove existing outputs.

The CLI stdout behavior is a compatibility change for any automation that parses the default JSON receipt. Mitigation:

- add `--json` before changing default stdout;
- document the change in README or release notes when implemented;
- keep the receipt shape stable in `--json` mode.

`src/report/generate-report.js` may need a decision: either remain JSON/Markdown-only for fixture workflows or gain methodology/CSV output options. The implementation should preserve fixture determinism and avoid surprising extra fixture writes unless explicitly requested by tests or commands.

Before M2 activation, the implementation packet should decide whether `src/report/generate-report.js` participates in CSV/methodology artifact generation. The conservative default is to keep it backwards-compatible and JSON/Markdown-only unless explicit flags are added.

## Failure Modes

- CSV write fails after core artifacts are staged:
  - The artifact transaction should roll back or fail the run before final promotion.
- `--no-csv` is passed:
  - Core artifacts and methodology should still be written unless methodology is intentionally tied to CSV scope.
- `--json` is passed:
  - The full receipt should be valid JSON on stdout; progress and caveats must not be interleaved with stdout.
- Sensitivity analysis cannot be computed because required PR metrics are missing:
  - The report should omit sensitivity details or label them unavailable rather than crashing.
- Methodology artifact generation fails:
  - Treat as an artifact generation failure, not a successful analysis, unless explicitly designed as best-effort.

## Security / Safety Considerations

- CSVs may contain private repository names, PR numbers, PR titles, URLs, file-role summaries, and count metadata. This is less raw than the source bundle but still sensitive.
- Artifact sensitivity warnings should name CSVs alongside source bundles, normalized data, metrics summaries, and reports.
- No generated artifact should include GitHub tokens or secret-bearing environment details.
- The report must continue avoiding individual contributor and reviewer rankings.
- `--no-csv` should be easy to find in CLI help for users who want fewer shareable artifacts.

## Validation

- Unit and golden tests for Markdown report layout and labels.
- Contract tests for CSV headers, row values, escaping, empty values, and excluded raw fields.
- CLI integration tests for default output, `--json`, `--no-csv`, artifact transactions, and failure paths.
- CLI `--json` tests that parse stdout as JSON and verify progress is not mixed into stdout.
- Manual live smoke test against `hannasdev/mcp-writing --limit 30`.
- Manual review that report conclusions can be traced from Markdown tables to CSVs, JSON report, metrics summary, and source bundle.
